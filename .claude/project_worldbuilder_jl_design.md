Design for `WorldBuilder.jl`, step 2 of [[project_gwb_julia]] (step 1, `WorldBuilder_jll`, has a PR open on Yggdrasil — see that memory for status). This package builds `.wb` JSON files from Julia and calls the GWB binaries/C API to generate output (e.g. ParaView `.vtu` via `gwb-grid`).

**Why these choices matter:** GWB ships a complete, mechanically-generated JSON Schema at `doc/world_builder_declarations.schema.json` (~19k lines, in the GWB repo at `/Users/kausb/Software/GeodynamicWorldBuilder/WorldBuilder`) describing every feature/sub-model/parameter via standard JSON Schema (`oneOf`, `enum`, `default value`, `description`, types). It's auto-regenerated by GWB's own test suite (`tests/gwb-dat/generate_declarations_and_schema`), so it stays in sync with the C++ source — but it is a generated artifact, **verify it still exists at that path and is current before relying on it again**, since GWB's schema can change between versions and the file path/test name could be renamed upstream.

Schema shape (as of the version reviewed): 6 feature types (`continental plate`, `oceanic plate`, `mantle layer` — simple polygon + depth range; `subducting plate`, `fault` — polyline + dip + segments/sections; `plume` — point-like), ~5-6 sub-model categories per feature (temperature/composition/grains/velocity/density models, +topography for plate features), ~18 unique model plugin names reused across features (uniform, linear, adiabatic, half space model, plate model, mass conserving, gaussian, perlin_noise, euler pole, etc.), ~85 total (feature × category × model) parameter-set combinations.

**Decisions made (2026-06-25):**
1. **Schema-driven codegen**: write a one-time/regenerate-on-demand codegen script that walks the JSON Schema and emits typed Julia structs/builder functions for every feature and sub-model, with docstrings/defaults pulled from the schema. Hand-write only the parts the schema doesn't fully capture: top-level `World`/file scaffolding, segment/section nesting + interpolation semantics for subducting plate/fault, and coordinate-system-dependent constraints.
2. **Typed structs**, not Dict/JSON3 builders — each feature/model gets a Julia struct with typed keyword-constructed fields and schema-matched defaults, serialized to the `.wb` JSON shape via a generic `to_dict`-style step. Chosen for REPL discoverability/autocomplete and compile-time field checking over Dict flexibility.
3. **Each pipeline step exposed separately** (not one mega `visualize()` call): something like `write_wb(world, path)`, `write_grid_config(cfg, path)`, `run_gwb_grid(wb_path, grid_path)` as composable building blocks — user wires the pipeline themselves. Chosen over a single high-level function for scriptability and reuse (e.g. one `.wb` file across multiple grid configs).

**How to apply:** when implementing, start with the codegen script (parses the schema, emits struct defs) before hand-writing any model-specific code — most of the ~85 combinations should fall out of that for free. Use [[project_gwb_julia]] for the broader two-step plan and Yggdrasil/JLL status context. Validate generated `.wb` output against GWB's own `tests/`/`cookbooks/` fixtures (run them through the real `WorldBuilder_jll` binaries) rather than hand-checking JSON structure.

**Additional requirements decided (2026-06-25, same session):**

4. **Version tracking, end to end.** Every generated file gets a provenance header (GWB version + commit + schema file hash + generation date). Package-level constants `GWB_SCHEMA_VERSION`/`GWB_SCHEMA_COMMIT` exposed for introspection. A runtime check (e.g. in `__init__`) compares `GWB_SCHEMA_VERSION` against `pkgversion(WorldBuilder_jll)` and `@warn`s on mismatch (warn, not error, since most GWB schema changes are additive). `gen/schema_cache/` snapshots are named per-version (e.g. `v1.1.0_world_builder_declarations.schema.json`), not overwritten, so history is diffable across GWB bumps. `Project.toml` carries an explicit `WorldBuilder_jll` compat bound, bumped deliberately alongside each schema regeneration.

5. **Read existing `.wb` files, not just write them.** Codegen emits `from_dict(::Type{T}, d)` alongside `to_dict(x)` for every generated struct (dispatch on the `"model"` key within each `oneOf` category). Public API: `read_wb(path) -> World` round-trips with `write_wb`. This unlocks editing existing cookbooks/tutorials from Julia (load → tweak fields → re-save) and migration for users with existing hand-written `.wb` files, and turns the planned round-trip validation (point 5 of the original plan) into `read_wb(fixture) |> write_wb |> diff-via-gwb-dat`, a stronger end-to-end test than generating from scratch alone. Design wrinkle: `from_dict` should be **lenient** on unknown keys (warn + stash in a catch-all field) rather than erroring, since a `.wb` from a different GWB version than our pinned schema may have fields we haven't generated structs for — this doubles as a concrete trigger for "the schema cache needs regenerating," tying back into point 4.

6. **Custom REPL printing (`show` methods)**, generated not hand-written per type — one reflective `Base.show(io, ::MIME"text/plain", x)` method per *abstract* category supertype (`AbstractFeature`, `AbstractTemperatureModel`, etc.), not 85 bespoke methods. Three display levels: (a) single-line summaries for leaf sub-models showing type name + key params, (b) compact multi-line tree per feature showing geometry size + nested sub-models by category/count rather than full field dumps, (c) a top-level `World` summary (schema version, coordinate system, numbered feature list) so "what am I building" is answerable at a glance instead of reading a raw struct dump.

**Implementation status: built and working (2026-06-25).** Package created at `~/.julia/dev/WorldBuilder.jl` (not under `~/Documents/GitHub` — user corrected this; dev packages live in `~/.julia/dev` in this setup, matching `LaMEM.jl`/`PETSc.jl`/etc.). 65/65 tests pass, including real integration tests against actual `WorldBuilder_jll` binaries (gwb-dat/gwb-grid producing byte-identical numeric output vs. hand-written reference `.wb` files, and real ParaView `.vtu` output).

Key files: `gen/generate.jl` (codegen, parses `gen/schema_cache/v1.1.0_world_builder_declarations.schema.json`, emits `src/generated/models.jl`), `src/core.jl` (World/coordinate systems/gravity model, `read_wb`/`write_wb`, strips GWB's `//`-style JSON comments before parsing), `src/segments.jl` (Segment/Section typed wrappers for SubductingPlate/Fault, since segments/sections aren't expressible by the flat per-(feature,category,model) codegen pattern), `src/show.jl` (the reflective show methods), `src/run.jl` (WorldBuilder_jll wrappers + `use_local_build!`/`use_jll!` to swap in a locally-compiled GWB build).

Real bugs found and fixed during implementation (relevant if regenerating or extending the codegen later):
- JSON3 parses whole-number JSON floats (e.g. schema default `0.0`) back as `Int64`, not `Float64` — codegen must force `Float64(x)` for all numeric literals or `Union{Float64,...}`-typed field defaults fail to construct.
- The `min depth`/`max depth`-style schema idiom (`oneOf[number, complex per-coordinate-override structure]`) sometimes has a 3rd `oneOf` branch (a string), not just 2 — codegen's pattern-match needed to handle `oneOf` length >= 2, not == 2. These fields are typed `Any` (not `Union{Float64,Vector{Any}}`, which Julia collapses to `Any` anyway) since the override structure's nesting is deep/version-unstable; only the common Float64 case is type-hinted via docstring.
- The same (category, model) pair (e.g. "uniform" temperature) has **different field sets per owning feature** (e.g. `min depth` for continental plate vs. `min distance slab top` for subducting plate) — codegen generates one struct per `(feature, category, model)` triple, not per `(category, model)`, and `from_dict` dispatch for sub-models is keyed on `(owner_feature, category, model_name)`, not just `model_name`.
- `gwb-grid` has no output-path CLI flag — it always writes `<wb-basename>.(p)vtu` into its current working directory, so `run_gwb_grid` must `cd` (via `Cmd(...; dir=...)`) rather than pass a path argument.
- `ccall((:fn, libpath), ...)` requires `libpath` to be a compile-time constant or *global* variable — a local/function-argument variable is a hard syntax error. Since the local-build-vs-JLL library path is chosen at runtime, point-query functions use `Libdl.dlopen`/`dlsym` instead (with a small handle cache keyed by path, invalidated on `use_local_build!`/`use_jll!`).
- `Pkg.test()`/sandboxed environments fail on the unregistered `WorldBuilder_jll` dependency (same issue hit during JLL packaging — see [[project_gwb_julia]]); run `julia --project=. test/runtests.jl` directly instead until the JLL is registered.
- GWB `.wb` files commonly contain `//`-style line comments (rapidjson is permissive; standard JSON/JSON3 is not) — `read_wb` strips them with a string-boundary-aware stripper before calling `JSON3.read`.

**How to apply (updated):** the design is no longer prospective — treat this as the current architecture. When asked to extend feature/model coverage, regenerate via `gen/generate.jl` first and only hand-write what doesn't fall out of that. When local GWB binaries exist (commonly at `/Users/kausb/Software/GeodynamicWorldBuilder/WorldBuilder/bin` after a local CMake build with `-DWB_ENABLE_APPS=ON`), `WorldBuilder.use_local_build!(dir)` lets testing/dev proceed without needing the registered JLL; this only requires the executables (gwb-dat/gwb-grid), not a shared `libWorldBuilder` — the dylib-dependent point-query path (`load_world`/`gwb_temperature`/`gwb_composition`) is optional and errors clearly if unavailable rather than blocking the executable-based path.
